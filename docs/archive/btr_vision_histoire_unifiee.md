## 1. Manifeste & Philosophie BTR

### La conviction fondatrice

Bad Technology Research est né d'une conviction simple : **l'intelligence artificielle sans compréhension humaine n'est que du calcul. La psychologie sans technologie reste limitée.** La technologie la plus puissante est celle qui comprend ce que l'humain ressent.

BTR ne construit pas pour impressionner. BTR ne construit pas pour vendre vite. BTR construit pour **durer**.

### Philosophie

- **L'IA a besoin de psychologie, pas seulement de capacités.** Un agent qui comprend les émotions, les patterns de pensée et les besoins profonds de son utilisateur est infiniment plus précieux qu'un modèle plus rapide ou plus gros.
- **L'humain au centre.** Chaque décision technique est évaluée sur son impact humain, pas seulement sur ses performances.
- **La contrainte est un avantage.** Développé depuis un smartphone, le code est radicalement simple, lisible, maintenable.
- **Zéro magie.** Pas de framework lourd, pas d'abstraction inutile.

### Taglines

| Niveau | Phrase |
|--------|--------|
| Tagline global BTR | *"For the ones who feel."* |
| Tagline Santana | *"Intelligent. Profond. Humain."* |
| Phrase éditoriale | *"La mémoire vivante."* |
| Manifeste | *"La technologie paraît ancienne. Le futur paraît nostalgique. Nous construisons la mémoire vivante."* |

## 4. Santana — L'Agent & Sa Vision

### Ce qu'est Santana

Santana n'est pas un outil. C'est une **présence**. Il écoute, il comprend, il se souvient, il grandit avec vous. Santana est le premier agent BTR — la fondation personnelle de Serge.

### Les Trois Piliers

- **Intelligent** — raisonne, apprend, évolue
- **Profond** — comprend, pas seulement répond
- **Humain** — ressent ce que l'humain ressent

### Pourquoi Investir dans l'Approche BTR

**Le constat :** L'industrie construit des modèles toujours plus gros, plus chers, plus génériques. BTR prend le chemin inverse : des agents légers, profonds, personnalisés.

**L'avantage BTR :**

- **Approche unique :** La psychologie appliquée à l'IA. Santana est un compagnon cognitif qui construit une mémoire partagée.
- **Contrainte = avantage :** Code radicalement simple, pas de dette technique inutile.
- **R&D frugale :** Stack à ~3.33€/mois. Pas de GPU, pas de cloud coûteux.
- **Architecture extensible :** Noyau factorisable en YAML. Un noyau, des personnalités multiples.

**Feuille de route produit :**

| Période | Étape |
|---------|-------|
| T3 2026 | Santana V3 Final — Migration Oracle, mémoire vectorielle |
| T4 2026 | Santana Marketing (Beta) — création de contenu, analyse de marché |
| T1 2027 | OpenClaw Platform — place d'agents spécialisés |
| T2 2027 | Licences Enterprise — agents sur mesure pour entreprises |

**Positionnement :** Cible le conseil tech, les startups IA, les entreprises, le contenu technique. GitHub = preuve, pas vitrine.

## 8. Décisions Marquantes

### D1 : DeepSeek plutôt qu'OpenRouter/Groq/Gemini

**Contexte :** OpenRouter trop cher, Groq quotas épuisés, Gemini quotas épuisés, Mistral qualité insuffisante.
**Décision :** DeepSeek V4 Flash direct (api.deepseek.com, modèle deepseek-chat).
**Impact :** Meilleur ratio qualité/coût. Invariant : chaque message doit avoir "role" explicite.

### D2 : Telegram plutôt que PWA

**Contexte :** La Forge (Flask PWA) construite puis supprimée le 16/05/2026.
**Décision :** Interface 100% Telegram. Commandes /help, /statut, /lire, etc.
**Impact :** Plus rapide, plus fiable, zéro maintenance web. Serge utilise Santana depuis un smartphone.

### D3 : sentence-transformers plutôt que TF-IDF

**Contexte :** TF-IDF testé en premier — pertinence ~7-8%.
**Décision :** all-MiniLM-L6-v2 (384 dims, 192MB RAM CPU). Pertinence 31%.
**Impact :** torch CPU suffisant. Version GPU désinstallée.

### D4 : Suppression des Rituels et Disciplines

**Contexte :** Trop complexes pour l'usage réel. Non utilisés par Serge.
**Décision :** Code supprimé le 16/05/2026. Ne pas recréer.
**Impact :** Surface de bug réduite, code plus simple.

### D5 : Architecture "Immutable Core"

**Contexte :** Le code évoluait trop vite, chaque modification risquait un bug.
**Décision :** Moteur Python petit (~400 lignes) et quasi immuable. Logique variable externalisée en Markdown (soul/SOUL.md, skills/*.md, etc.).
**Impact :** Modifier le comportement = éditer un fichier texte, pas toucher au code.

### D6 : Migration Oracle — Deadline non négociable

**Contexte :** GCP Cloud Shell expire le 11/07/2026. Disque 48G structurellement limité.
**Décision :** Migration vers Oracle Free Tier ARM en 3 sessions (~5h). Rien d'autre avant.
**Impact :** Risque opérationnel majeur si non respecté.

## 9. Leçons Apprises

### Pièges Évités

- **KISS Applied :** Garder le moteur petit et stable évite les bugs en cascade.
- **Sécurité proactive :** .env protégé, TokenFilter — plutôt que de réagir à une fuite.
- **Mémoire vectorielle :** TF-IDF (7%) → sentence-transformers (31%) a transformé la pertinence mémoire.
- **Mobile-first :** L'architecture pensée pour smartphone a forcé une simplicité radicale.

### Erreurs Coûteuses

- **PWA construite trop tôt :** La Forge a coûté des semaines de développement pour être finalement abandonnée. La leçon : valider l'usage avant de construire l'interface.
- **Documentation tardive :** Les ADR et DECISIONS.md n'ont été créés qu'en Cycle 3. Beaucoup de contexte a été perdu entre les phases 0-4.
- **Tests trop tardifs :** Les tests unitaires sont arrivés au Cycle 3. Avant, chaque déploiement était un risque.

### Principes Établis

1. **Une modification = un test = un snapshot.** Dans cet ordre, sans sauter d'étape.
2. **Ne pas enchaîner deux modifs sans avoir sauvegardé la première.**
3. **Ne jamais déployer sans validation syntaxique.**
4. **Ne jamais recréer ce qui a été supprimé** (Rituels, Disciplines, PWA, generate_pdf).
5. **Documenter la décision AVANT d'implémenter.**
